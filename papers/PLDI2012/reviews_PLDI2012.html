This paper discusses techniques for (more precise) path-by-path
abstract interpretation of numerical program properties. In the first
part (sections 3 and 4), the paper discusses how to combine algorithms
from [16] and [26], and then [17]. In the second part of the paper,

The combination of the techniques of [16], [26] and [17] is
interesting, although rather unsurprising.

The experimental evaluation is also interesting, although the overall
results are again unsurprising (the more precise combinations, like
G+PF and DIS, are also the more expensive -- see Table 3). As
acknowledged by the authors in Section 6, the experimental setup

points) is perhaps questionable -- overall, what can we take away from

The presentation looks good, but I didn't check all the details. The
experimental section was a bit harder to read.

Overall, all this makes sense, at a high level. This looks like a
decent paper, but its topic and results aren't the most exciting to
me.

          >>> Summary of the submission <<<

The paper takes two existing analyses -- (1) Guided Static Analysis, a
technique to improve widening precision by incrementally processing
increasing subsets of program paths; and (2) Path Focusing, a
technique for path-sensitive program analysis based on SMT solvers --
and combines them into a new analysis that uses SMT solving to
incrementally process increasing subsets of executable program paths.

The paper empirically evaluates the new technique against the two
existing techniques, showing increased precision while remaining
within the same order of magnitude of performance. It also evaluates
the effect on precision of varying the abstract domain of the analysis
while keeping the analysis technique constant.



          >>> Evaluation <<<

The paper is well written and makes a solid contribution. I was not
familiar with guided static analysis or path focusing before now
(though the background section did a good job explaining them); my
understanding of the paper is:

1. The problem with guided static analysis is that it will eventually
process all paths in the control-flow graph, including non-executable
paths, thus losing precision.

2. The problem with path focusing is that (1) it will process all
executable paths at once, including potentially many redundant paths
that don't contribute to the solution; and (2) its widening operation
lacks the precision conferred by guided static analysis.

Therefore, by combining these two techniques we get an analysis that
will only examine 'interesting' executable paths that will contribute
to the analysis solution, and one that still benefits from the
widening procedure of guided static analysis, thus getting the best of
both worlds.

The benefits are empirically evaluated in terms of numerical abstract
domains (intervals, convex polyhedra, etc). It would be interesting to
see how much benefit this technique confers to other types of abstract
domains, including those that don't require widening (e.g., pointer
analysis). This possible use is only briefly mentioned at the very end
of the paper; I'd like to know more about how applicable the technique
is to other kinds of abstract domains (e.g., is it a straight-forward
extension or does it require further research? How much benefit might
be expected?).

Empirical evaluation:

-- Since the implemented analyses are all intraprocedural, it would be
  useful to know something about the sizes of the functions in the

-- I was expecting that a conclusion would be drawn based on the
  empirical results about the relative merits of increasing precision
  using more precise techniques or more precise abstract
  domains. Instead, these two questions are treated completely
  independently. The evaluation of different abstract domains thus
  feels rather tacked on and divorced from the rest of the paper.

Some notes about mistakes/ambiguities:

          >>> Summary of the submission <<<

Abstract interpretation is a continuous struggle to finding the sweet
spot between precision and performance and scalability. Recent
advancements on SMT solving, guided static analysis, and path-focusing
techniques, the many-fold abstract domains they can be combined with
including intervals, octagons, and polyhedra to quote only those
advancements being relevant for the present submission make the
informed choice of which technique to apply when for a given set of
requirements on precision and performance ever more difficult.

The present paper aims at improving on this situation by (re-)
implementing and integrating SMT solving, guided static analysis, and
path-focusing techniques in a uniform prototype tool and comparing
then the efficacy of various combinations thereof on a set of open
source code projects like grep, gzip, tar, and others. Additionally,
it presents and investigates a new variant combining preferable
features from guided static analysis and path-focusing technique.


          >>> Evaluation <<<

In principle, I like the approach presented in this paper very
much. The author(s) made quite a good job in making a lot of
state-of-the-art analysis techniques for computing disjunctive
invariants available in a form allowing to compare their relative
efficacy in the same setting. I am bit disappointed, however, about
the conclusions drawn from the many practical experiments. Of course,
it were naive to expect as an outcome of these experiments a total
ordering between the various approaches. However, a set of guidelines
suggesting the usage of one approach over another in terms of if-what
scenarios would be very desirable. Maybe the differences observed
between the various approaches (usually some 2 or 5% difference) are
too small in order to allow for such rules, and might also be too
small in order to trace the reasons for these differences back to a
set of roots. Without something like that, however, the numbers
obtained in the experiments are of limited usability. 2 to 5%
difference, does it make a difference at all? If so, for which
problems or e.g. transformations based on top of these analysis
results? As currently presented, the findings and conclusions drawn
from them do not really help.

A problem of the presentation might also be that on the one hand side
the paper aims at comparing a set of competing state-of-the-art
approaches, on the other hand, it proposes a new variant being a
mixture of features of those and including it in the comparison.
Maybe this is too much for a single paper.

Last but not least, is the title of the paper accurate? From the
abstract and introduction, I would have expected something like
Disjunctive program analysis: When to use which technique why?

Nonetheless, the paper makes an interesting contribution.
